# 自助问答到人工处理闭环业务流程

## 1. 业务概述与背景

自助问答到人工处理闭环是 OpsMind 运维数字员工系统的核心业务流程。该流程面向报障人和一线员工，通过门户问答降低重复咨询成本；当 AI 无法解决问题时，系统自动引导用户提交申告，并由后台运维人员完成人工处理、回访和知识沉淀。

流程覆盖门户智能问答、申告创建、后台人工处理、回访记录、知识候选生成、知识审核发布和后续 RAG 问答复用。相关接口定义见 `docs/API/portal/chat.md`、`docs/API/portal/tickets.md`、`docs/API/admin/tickets.md`、`docs/API/admin/knowledge-candidates.md`、`docs/API/admin/knowledge-articles.md`。

## 2. 适用范围与业务边界

适用范围：

| 范围 | 说明 |
|------|------|
| 门户智能问答 | 报障人提交运维问题，系统同步返回答案、来源和置信度 |
| 门户申告 | 问答未解决或用户主动报障时创建申告 |
| 后台处理 | 运维人员查看、处理、更新申告状态并填写处理记录 |
| 回访闭环 | 运维人员对已处理申告登记回访结果 |
| 知识沉淀 | 已完成申告可生成知识候选，经审核发布后进入知识库 |

业务边界：

| 边界项 | 约束 |
|--------|------|
| AI 问答 | MVP 阶段使用同步响应，不包含 SSE 流式输出 |
| RAG 编排 | 由 AnythingLLM 负责完整 RAG 流程，后端通过 `RagClient` 适配 |
| 模型推理 | 通过 vLLM OpenAI-compatible 适配层调用 |
| 附件上传 | 门户匿名附件上传仅允许申告场景使用 `POST /api/v1/portal/files` |
| 知识发布 | 发布后通过 NATS JetStream 触发 Worker 异步同步 |

## 3. 参与角色与职责定义

| 角色 | 职责 |
|------|------|
| 报障人 | 在门户提交问题、查看答案、提交反馈、创建申告、查询处理进度 |
| AI 问答服务 | 调用 AnythingLLM 和 vLLM，返回答案、来源、置信度和转人工状态 |
| 运维人员 | 查看申告、更新状态、填写处理过程和结果、登记回访、生成知识候选 |
| 知识库管理员 | 审核知识候选转化结果，维护知识条目并发布 |
| 系统管理员 | 配置模型、知识库、权限和审计策略 |
| Worker | 消费知识发布事件，同步 AnythingLLM 并写入 pgvector 切片向量 |

## 4. 整体业务流程总图说明

```mermaid
flowchart TD
    A[报障人进入门户] --> B[提交运维问题]
    B --> C[POST /api/v1/portal/chat-sessions]
    C --> D[Go API 写入 portal_chat_session]
    D --> E[调用 RagService.Chat]
    E --> F[AnythingLLM 检索与 RAG 编排]
    F --> G[vLLM 生成答案]
    G --> H{答案是否可用}
    H -->|可用| I[门户展示答案和来源]
    I --> J{用户反馈}
    J -->|resolved| K[POST /api/v1/portal/chat-sessions/{session_no}/feedback]
    J -->|unresolved| L[进入申告创建]
    H -->|不可用| L
    L --> M[可选上传附件 POST /api/v1/portal/files]
    M --> N[POST /api/v1/portal/tickets]
    N --> O[写入 ticket 和附件绑定]
    O --> P[运维人员 GET /api/v1/admin/tickets]
    P --> Q[PATCH /api/v1/admin/tickets/{id}/status]
    Q --> R{是否处理完成}
    R -->|否| P
    R -->|是| S[POST /api/v1/admin/tickets/{id}/visit-records]
    S --> T[POST /api/v1/admin/tickets/{id}/knowledge-candidate]
    T --> U[POST /api/v1/admin/knowledge-candidates/{id}/convert]
    U --> V[提交审核和发布知识]
    V --> W[Worker 同步 AnythingLLM 和 pgvector]
    W --> X[新知识参与后续问答]
```

## 5. 分步详细流程

| 步骤 | 操作 | 责任人/系统 | 输入 | 输出 |
|------|------|-------------|------|------|
| 1 | 打开门户问答页 | 报障人 | 运维问题 | 问题文本 |
| 2 | 提交问答请求 | 前端 `web` | `question`、`knowledge_base_code`、`reporter_contact` | `POST /api/v1/portal/chat-sessions` |
| 3 | 保存问答会话 | Go API `server` | 问答请求 | `portal_chat_session`、`portal_chat_message` |
| 4 | 调用 RAG 服务 | `RagService`、`RagClient` | 问题、知识库编码、上下文 | AnythingLLM 请求 |
| 5 | 生成答案 | AnythingLLM、vLLM | 检索结果、提示词、模型参数 | 答案、来源、置信度 |
| 6 | 返回问答结果 | Go API | RAG 响应 | `status=answered`、`need_ticket` 或 `failed` |
| 7 | 提交问答反馈 | 报障人 | `resolved` 或 `unresolved` | `POST /api/v1/portal/chat-sessions/{session_no}/feedback` |
| 8 | 上传申告附件 | 报障人 | 截图、PDF、说明文档 | `POST /api/v1/portal/files` 返回 `file_id` |
| 9 | 创建申告 | 报障人 | 联系方式、标题、描述、紧急程度、附件 ID | `POST /api/v1/portal/tickets` |
| 10 | 初始化申告 | Go API | 申告请求 | `ticket`、附件绑定、初始状态 `pending` |
| 11 | 查询待处理申告 | 运维人员 | 状态、紧急程度、关键词 | `GET /api/v1/admin/tickets` |
| 12 | 处理申告 | 运维人员 | 状态、处理过程、处理结果 | `PATCH /api/v1/admin/tickets/{id}/status` |
| 13 | 登记回访 | 运维人员 | 回访结果、回访内容 | `POST /api/v1/admin/tickets/{id}/visit-records` |
| 14 | 生成知识候选 | 运维人员 | 已完成申告 | `POST /api/v1/admin/tickets/{id}/knowledge-candidate` |
| 15 | 转为知识条目 | 知识库管理员 | 候选内容、目标知识库、分类 | `POST /api/v1/admin/knowledge-candidates/{id}/convert` |
| 16 | 审核并发布知识 | 知识库管理员 | 知识条目 | `submit-review`、`review`、`publish` |
| 17 | 同步知识 | Worker | `knowledge.published` 事件 | AnythingLLM 同步记录、`knowledge_chunk` |

## 6. 异常分支与兜底处理流程

| 异常场景 | 触发条件 | 处理流程 | 结果 |
|----------|----------|----------|------|
| 问题为空或超长 | `question` 不满足校验 | 返回 `validation_error`，前端提示修正 | 不创建会话 |
| 知识库不存在或停用 | `knowledge_base_code` 无效 | 返回 `knowledge_base_not_found`，可改用默认知识库 | 不继续 RAG 调用 |
| AnythingLLM 不可用 | RAG 服务连接失败或超时 | 返回 `rag_unavailable`，前端展示转人工入口 | 用户可创建申告 |
| vLLM 不可用 | 模型服务失败或超时 | 返回 `model_unavailable`，前端展示转人工入口 | 用户可创建申告 |
| 匿名请求过频 | IP 超过限流阈值 | 返回 `rate_limit_exceeded` | 用户稍后重试 |
| 附件不存在 | `attachment_ids` 中存在无效 ID | 返回 `file_not_found` | 用户重新上传附件 |
| 联系方式不匹配 | 门户查询申告时手机号不一致 | 返回 `contact_mismatch` | 拒绝展示申告详情 |
| 状态流转非法 | 从当前状态流向不允许状态 | 返回 `invalid_status_transition` | 运维人员刷新详情后重试 |
| 重复生成候选 | 同一申告已存在候选 | 返回 `conflict` 或 `invalid_ticket_status` | 进入已有候选处理 |
| 知识同步失败 | Worker 调用 AnythingLLM 或 pgvector 写入失败 | 写入 `knowledge_sync_record`，更新 `rag_sync_status=failed` | 管理员查看错误并重试 |

## 7. 准入条件 / 准出条件

准入条件：

| 条件 | 说明 |
|------|------|
| 门户可访问 | 前端门户页面正常加载 |
| 默认知识库可用 | `knowledge_base.status=active`，embedding 配置完整 |
| 模型和 RAG 服务可用 | AnythingLLM、vLLM 连接配置有效 |
| 附件存储可用 | MinIO 或 S3-compatible 服务配置有效 |
| 后台权限配置完成 | 运维人员具备 `ticket:list`、`ticket:process` 等权限 |

准出条件：

| 条件 | 说明 |
|------|------|
| 已解决问答有反馈 | `portal_chat_feedback.feedback_type=resolved` |
| 未解决问题有申告 | `ticket.status` 进入 `pending` 或后续处理状态 |
| 人工处理有记录 | `ticket_process_record` 记录处理过程和处理结果 |
| 已完成申告可追溯 | 申告详情包含问答上下文、附件、处理记录和回访记录 |
| 有效方案可沉淀 | `knowledge_candidate` 可转为 `knowledge_article` |
| 发布知识可复用 | `knowledge_article.status=published` 且同步成功 |

## 8. 业务规则、约束与风险点

业务规则：

| 规则 | 说明 |
|------|------|
| 问答状态枚举 | `processing`、`answered`、`need_ticket`、`failed` |
| 申告状态流转 | `pending -> processing -> need_more_info/completed -> closed` |
| 处理完成要求 | 状态改为 `completed` 时必须填写 `process_result` |
| 附件绑定 | 门户附件上传后在申告创建成功时绑定到 `ticket` |
| 候选唯一性 | 一个申告最多生成一个知识候选 |
| 知识发布要求 | 知识必须先审核通过，再发布进入同步流程 |
| 向量一致性 | 同一知识库下 embedding 模型和向量维度必须一致 |
| 审计追踪 | 状态变更、知识发布、权限变更等敏感操作必须写入审计日志 |

风险点：

| 风险 | 影响 | 控制措施 |
|------|------|----------|
| AI 回答不准确 | 用户误操作或重复申告 | 展示来源、置信度和转人工入口 |
| 匿名接口被滥用 | 资源消耗和垃圾申告 | IP 限流、附件大小限制、审计记录 |
| RAG 或模型不可用 | 问答不可用 | 降级为人工申告入口 |
| 状态流转混乱 | 处理责任不清 | 后端强制状态机校验 |
| 知识同步失败 | 新知识无法参与问答 | 同步记录、失败状态、人工重试 |
| 附件泄露 | 用户敏感信息外泄 | 预签名短链、权限校验、对象路径隔离 |
