## 1. 概述

企业运维工作存在重复性高、流程性强、知识依赖重、人工响应压力大的特点。一线员工在账号管理、常见问题咨询、故障申告、报障跟进等场景中，需要反复查询制度、知识库和处理流程，导致响应效率低、服务体验不稳定、知识沉淀不足。

本项目基于 AI、RAG 私有知识库和自动化运维能力，建设面向企业运维场景的数字员工系统。系统通过本地化大模型、私有知识库、运维申告门户和后台运营管理能力，辅助或替代人工完成常见咨询、自助处理、申告记录、人工流转和知识库更新等工作。

### 项目特色

- **私有化部署：** 大模型与知识库优先支持本地部署，适配企业数据安全要求。
- **RAG 知识增强：** 基于企业运维知识库提供可追溯、可维护的问答能力。
- **人机协同闭环：** 机器人无法处理时流转人工，人工处理结果可沉淀为知识。
- **运维门户一体化：** 前台服务入口与后台维护能力统一建设。
- **企业级架构取向：** 后端优先采用 Go，前端优先采用 Vue，架构参考 GVA 项目的前后端分离设计思路。

### 用户类型

| 用户类型 | 用户特性 | 核心诉求 |
| --- | --- | --- |
| 报障人/一线员工 | 对运维流程不熟悉，期望快速获得答案 | 自助查询、提交问题、查看处理结果 |
| 运维人员 | 负责处理账号、故障和申告事项 | 快速定位问题、记录处理过程、回访用户 |
| 知识库管理员 | 负责维护 FAQ、处理方案和知识文档 | 审核知识、更新内容、提升问答质量 |
| 系统管理员 | 负责用户、角色、权限和系统配置 | 账号管理、权限控制、运行监控 |
| 项目评审人员 | 关注课程设计完整性和系统可演示性 | 清晰业务流程、可运行原型、完整文档 |

| 角色 | 说明 | 典型权限 |

## 2. 目标

系统目标是为企业提供智能化运维服务入口，降低一线运维人员重复劳动，提高常见问题自助解决率，并形成可持续迭代的运维知识资产。

- 提供常见运维问题的自助问答和处理指引。
- 支持无法自动解决的问题进入在线记录或人工处理流程。
- 支持运维账号增删改查等后台维护能力。
- 支持人工处理结果回流知识库，提升后续自动应答质量。

## 3. 用户故事

### US-001: 智能问答

**描述：** 作为报障人，我希望输入运维问题后立即获得处理方案，以便不用等待人工响应。

**流程：**

1. 用户进入运维门户。
2. 用户输入问题，例如"账号冻结怎么处理"。
3. 前端提交问题到后端问答接口。
4. 后端通过 AnythingLLM 适配层调用知识检索服务获取相关 FAQ 或文档片段。
5. 后端通过 vLLM OpenAI-compatible 适配层调用模型服务生成答案。
6. 前端同步展示完整答案、来源和"仍需人工处理"入口。
7. 用户确认解决或转入申告流程。
8. **降级流程：** 若 AnythingLLM 或 vLLM 服务不可用，后端返回明确提示"当前 AI 服务暂不可用，请提交申告由人工处理"，前端展示该提示并直接引导进入申告提交。

**界面交互：**

- 问答输入框支持多行输入和提交按钮。
- 回答区域展示处理步骤、来源文档名称、时间和反馈按钮。
- 答案反馈包含"已解决""未解决"。
- 未解决时展示申告入口，并自动带入原始问题与问答上下文。

**验收标准：**

- [ ] 用户提交问题后，系统能返回答案或明确的兜底提示。
- [ ] 答案展示知识来源且展示命中文档名称。
- [ ] MVP 阶段问答接口同步返回完整答案；流式输出不作为本期验收项。
- [ ] 无法回答时，用户可以一键进入申告提交。
- [ ] 用户反馈结果可被系统记录。
- [ ] AI 服务不可用时，返回明确提示并引导转人工。

### US-002: 申告记录

**描述：** 作为报障人，我希望机器人无法回答时可以提交申告记录，以便问题不会中断。

**流程：**

1. 用户从问答页或门户入口进入申告表单。
2. 系统自动带入原问题和问答上下文。
3. 用户补充影响范围、联系方式和紧急程度。
4. 用户提交申告。
5. 系统生成申告单并展示当前状态。

**界面交互：**

- 表单必填项包括问题标题、问题描述、紧急程度、联系方式（手机号）。
- 表单选填项包括邮箱、影响范围、受影响系统。
- **影响范围**为下拉选择：个人、部门、全公司。
- **受影响系统**为多选标签，从系统管理员预设的系统列表中选择（如：OA 系统、邮箱系统、VPN、办公网络等）。
- **紧急程度**为下拉选择：低、中、高。
- **联系方式**格式：手机号（必填，11 位数字），邮箱（选填，标准邮箱格式）。
- 提交成功后展示申告编号和状态。
- 用户可按申告编号或手机号查询处理进度。

**验收标准：**

- [ ] 必填项缺失时不能提交，并提示具体字段。
- [ ] 手机号格式校验：必须为 11 位数字，否则提示"请输入正确的手机号"。
- [ ] 提交成功后生成唯一申告编号（格式：TK-YYYYMMDD-XXXX）。
- [ ] 申告状态初始为"待处理"。
- [ ] 申告详情中保留原始问答上下文。

### US-003: 人工处理与回访

**描述：** 作为运维人员，我希望查看待处理申告并记录处理结果，以便统一跟踪问题闭环。

**流程：**

1. 运维人员登录后台。
2. 运维人员查看待处理申告列表。
3. 运维人员打开申告详情，查看用户问题和机器人问答上下文。
4. 运维人员点击"开始处理"，申告状态变更为"处理中"。
5. 运维人员填写处理过程和处理结果。
6. 运维人员登记回访结果。
7. 系统将申告状态更新为"已解决"或"需补充信息"。
8. 运维人员可选择生成知识库候选条目。

**界面交互：**

- 列表支持按状态、紧急程度、创建时间、关键词筛选。
- 详情页展示申告信息、问答上下文、处理记录和状态流转时间线。
- 状态按钮包括"开始处理""需补充信息""处理完成"。
- 处理完成时可勾选"沉淀为知识库候选"。

**回访结果数据结构：**

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| 回访方式 | 下拉选择 | 是 | 电话、邮件、IM、当面 |
| 回访内容 | 多行文本 | 是 | 回访沟通的具体内容 |
| 是否解决 | 单选 | 是 | 是、否 |
| 备注 | 多行文本 | 否 | 补充说明 |

**申告状态机：**

```
待处理 ──→ 处理中 ──→ 已解决
  │         │   ↑        │
  │         │   └────────┘（补充信息后回到处理中）
  │         ↓
  │     需补充信息（可循环，最多 3 次）
  │         │
  ↓         ↓
已关闭     已关闭（超时 7 天未处理自动关闭）
```

**状态流转规则：**

| 操作 | 前置状态 | 目标状态 | 触发方 |
| --- | --- | --- | --- |
| 开始处理 | 待处理 | 处理中 | 运维人员 |
| 需补充信息 | 处理中 | 需补充信息 | 运维人员 |
| 补充信息 | 需补充信息 | 处理中 | 报障人 |
| 处理完成 | 处理中 | 已解决 | 运维人员 |
| 关闭 | 任意 | 已关闭 | 运维人员 / 系统自动（7 天超时） |

- **待处理：** 申告初始状态，等待运维人员认领。
- **处理中：** 运维人员已认领，正在处理。
- **需补充信息：** 运维人员需要报障人补充信息。系统发送站内消息通知报障人，门户页顶部展示提示条"您有 X 条申告需要补充信息"。报障人补充后状态回到"处理中"。最多循环 3 次，超过后强制进入人工处理。
- **已解决：** 问题已处理完成。
- **已关闭：** 申告被手动关闭或超时自动关闭（创建后 7 天未处理）。

**验收标准：**

- [ ] 运维人员可以查看、筛选、处理申告。
- [ ] 每次状态变化均记录处理人和时间。
- [ ] 处理完成必须填写处理结果。
- [ ] 被标记为知识库候选的记录可进入知识审核流程。
- [ ] "需补充信息"状态触发站内消息通知报障人。
- [ ] 申告创建 7 天未处理自动变更为"已关闭"。

### US-004: 运维账号管理

**描述：** 作为系统管理员，我希望管理运维人员账号的新增、冻结、修改、查询，以便保障系统访问安全。

**流程：**

1. 管理员进入账号管理页面。
2. 管理员新增账号，填写姓名、账号、角色、手机号和初始密码。
3. 管理员可查询账号并查看详情。
4. 管理员可修改账号基础信息和角色。
5. 管理员可冻结或恢复账号。

**界面交互：**

- 账号列表支持关键词、角色、状态筛选。
- 新增和编辑使用弹窗或独立表单。
- 冻结账号需要二次确认并填写原因。
- 账号状态包括正常、冻结。

**密码策略：**

- 长度 8-32 位。
- 必须包含大写字母、小写字母和数字。
- 初始密码由管理员设置，用户首次登录后强制修改。
- 密码使用 bcrypt 哈希存储，禁止明文。

**验收标准：**

- [ ] 管理员可以完成账号新增、查询、修改、冻结。
- [ ] 账号名不可重复。
- [ ] 冻结账号无法登录后台。
- [ ] 敏感操作写入审计日志。
- [ ] 密码不符合策略时提示具体规则。
- [ ] 首次登录用户被强制引导到修改密码页面。

### US-005: 知识库管理

**描述：** 作为知识库管理员，我希望维护 FAQ、处理方案和文档资料，以便将人工处理后的有效方案沉淀到知识库，后续自动回答相同问题。

**流程：**

1. 管理员创建或选择知识库，并从系统预配置的 embedding 模型列表中选择模型和向量维度。
2. 管理员新增知识条目。
3. 管理员填写问题、答案、分类、标签和适用范围。
4. 管理员提交审核。
5. 审核人（知识库管理员，且不能是创建人本人）审核通过后发布知识。
6. 系统同步知识到 AnythingLLM，并将同模型同维度的切片向量写入 pgvector。

**界面交互：**

- 知识列表支持分类、状态、关键词筛选。
- 知识库配置支持从系统预配置的 embedding 模型和向量维度选项中选择。
- 编辑页支持 FAQ 结构化字段。
- 发布前需要审核。
- 已发布知识修改后需要重新审核。

**Embedding 模型配置：**

- 系统管理员在"系统配置 > AI 模型配置"中预配置可用的 embedding 模型列表。
- 每条配置包含：模型名称、模型类型（API 接入 / 本地部署）、API 地址或本地路径、向量维度。
- 知识库创建时，知识库管理员从该列表中选择一个模型和对应的向量维度。
- 同一知识库的所有知识切片必须使用相同的 embedding 模型和向量维度。

**审核规则：**

- 审核人必须具有"知识库管理员"角色。
- 审核人不能是知识条目的创建人本人（需另一人审核）。
- 审核结果：通过（进入发布流程）、驳回（退回修改，需填写驳回原因）。

**验收标准：**

- [ ] 知识条目可新增、编辑、审核、发布、停用。
- [ ] 同一知识库下所有向量必须使用相同 embedding 模型和维度。
- [ ] 已发布知识可被 AnythingLLM RAG 流程检索。
- [ ] 停用知识不再参与问答。
- [ ] 每个知识条目保留更新时间和维护人。
- [ ] 审核人与创建人不能是同一人。
- [ ] 驳回时必须填写驳回原因。

### US-006: 用户权限与登录

**描述：** 作为系统管理员，我希望管理登录认证、角色、菜单权限和接口权限，以便不同角色只能访问授权范围内的功能。

**流程：**

1. 用户输入账号密码登录系统。
2. 后端校验账号状态和密码。
3. 登录成功后返回访问令牌（JWT）和用户权限。
4. 前端根据权限展示菜单。
5. 后端对接口请求进行权限校验。

**界面交互：**

- 登录页包含账号、密码和登录按钮。
- 后台菜单按角色动态展示。
- 无权限访问时展示明确提示（403 页面或弹窗）。
- 管理员可维护角色和权限。

**验收标准：**

- [ ] 正常账号可登录，冻结账号不可登录。
- [ ] 不同角色看到不同菜单。
- [ ] 未授权接口请求被拒绝（返回 403）。
- [ ] 登录和退出记录可追踪。

### US-007: 数据看板

**描述：** 作为运维人员，我希望查看问答量、解决率、人工转派量和知识命中率，以便支持运营分析。

**指标定义：**

| 指标 | 计算公式 | 数据来源 |
| --- | --- | --- |
| 问答总量 | 统计周期内的问答会话总数 | chat_sessions 表 |
| 自动解决率 | 用户反馈"已解决"的问答数 / 问答总量 × 100% | chat_sessions 表 feedback 字段 |
| 人工转派量 | 由问答转入申告的记录数 | tickets 表（来源=问答转申告） |
| 知识命中率 | 返回了知识来源引用的问答数 / 问答总量 × 100% | chat_sessions 表 sources 字段非空 |

**界面交互：**

- 顶部统计卡片展示四项指标的当前值和环比变化。
- 中部折线图展示按日/周/月的趋势变化。
- 时间范围选择器：最近 7 天、最近 30 天、自定义范围。

**验收标准：**

- [ ] 展示问答总量、自动解决率、人工转派量、知识命中率。
- [ ] 支持按时间范围筛选（最近 7 天、30 天、自定义）。
- [ ] 数据可按日/周/月聚合展示。

**敏感操作清单：**

- 账号冻结/恢复

- 申告状态变更

- 系统配置修改

**验收标准：**

- [ ] 登录、退出操作记录日志。
- [ ] 冻结账号、发布知识、处理申告、角色修改、系统配置修改等敏感操作记录审计日志。
- [ ] 日志支持按操作类型、操作人、时间筛选。

### US-009: 模型配置

**描述：** 作为系统管理员，我希望配置模型地址、密钥、提示词和召回参数，以便适配不同环境。

**配置项：**

| 配置项 | 类型 | 说明 |
| --- | --- | --- |
| 模型服务地址 | URL | vLLM 或兼容 API 的服务地址 |
| API 密钥 | 密文字段 | 模型服务的认证密钥 |
| 系统提示词 | 多行文本 | 问答时注入的系统级提示词 |
| 知识检索 Top K | 数字（1-20） | 每次检索返回的相关片段数量，默认 5 |
| 置信度阈值 | 数字（0-1） | 低于此阈值的回答引导转人工，默认 0.6 |

**置信度说明：**

- 置信度由 AnythingLLM 在知识检索阶段返回，表示问题与命中文档的相似度分数（0-1）。
- 系统管理员可配置置信度阈值（默认 0.6）。
- 当所有命中文档的最高置信度低于阈值时，系统判定为"无法回答"，引导用户提交申告。
- 置信度记录在每次问答会话中，用于数据看板统计。

**验收标准：**

- [ ] 支持配置模型服务地址和 API 密钥。
- [ ] 支持配置系统提示词。
- [ ] 支持配置知识检索 Top K 值（1-20，默认 5）。

## 4. 功能需求

### P0 核心功能

- **FR-1：** 系统必须提供智能问答能力，用户输入运维问题后通过 RAG 知识检索和模型生成返回答案，答案包含处理步骤和知识来源文档名称。
- **FR-2：** 当智能问答无法解决问题（置信度低于阈值或无匹配知识）或 AI 服务不可用时，系统必须引导用户提交申告记录，自动生成唯一申告编号（格式：TK-YYYYMMDD-XXXX），状态初始为"待处理"。
- **FR-3：** 系统必须提供申告处理后台，运维人员可查看待处理申告、填写处理记录、登记回访结果，并将有效方案标记为知识库候选。
- **FR-4：** 系统必须提供运维账号管理能力，支持账号新增、查询、修改、冻结，冻结账号无法登录。密码策略：8-32 位，含大小写字母和数字，首次登录强制修改。
- **FR-5：** 系统必须提供知识库管理能力，支持知识条目的新增、编辑、审核、发布、停用，已发布知识可被 RAG 流程检索。审核人与创建人不能是同一人。
- **FR-6：** 系统必须提供用户权限控制，包括登录认证（JWT）、角色管理、菜单权限和接口权限，不同角色看到不同菜单，未授权请求返回 403。
- **FR-7：** 系统管理员必须能在系统配置中预配置可用的 embedding 模型列表（支持 API 接入和本地部署两种类型），每条配置包含模型名称、类型、地址/路径、向量维度。知识库创建时从该列表中选择，同一知识库下所有知识切片必须使用相同 embedding 模型和向量维度。
- **FR-8：** 智能问答链路必须通过 AnythingLLM 完整 RAG 流程检索和编排知识，并通过 vLLM OpenAI-compatible 适配层同步返回答案。

### P1 增强功能

- **FR-9：** 系统必须提供数据看板，展示问答总量、自动解决率、人工转派量、知识命中率，支持按时间范围筛选，数据可按日/周/月聚合。
- **FR-10：** 系统必须提供操作日志，记录登录日志和敏感操作审计（账号冻结/恢复、知识发布/停用、申告状态变更、角色权限修改、系统配置修改），每条日志包含操作人、操作时间、操作类型、操作对象、操作详情、IP 地址。
- **FR-11：** 系统必须提供模型配置能力，支持配置模型服务地址、API 密钥、系统提示词、知识检索 Top K（1-20，默认 5）和置信度阈值（0-1，默认 0.6），配置修改后即时生效。

### P2 扩展功能

- **FR-12：** 系统应预留智能巡检模块入口，支持巡检任务配置和巡检报告生成。
- **FR-13：** 系统应预留故障自愈模块入口，支持规则触发、脚本执行和结果回写。
- **FR-14：** 系统应预留告警中枢模块入口，支持告警接入、聚合、分派和闭环。

### 业务流程

#### 自助问答到人工处理闭环

1. 报障人进入门户并提交问题。
2. 系统检索知识库并生成回答。
3. 报障人反馈"已解决"时，系统记录问答效果。
4. 报障人反馈"未解决"或系统无法回答（置信度低于阈值）时，进入申告提交。
5. 运维人员在后台处理申告并回访用户。
6. 处理完成后，运维人员将有效处理方案标记为知识库候选。
7. 知识库管理员审核并发布知识。
8. 新知识参与后续智能问答。

#### 运维账号管理流程

1. 管理员创建运维账号并分配角色。
2. 运维人员使用账号登录后台，首次登录后强制修改密码。
3. 管理员根据人员变化修改或冻结账号。
4. 系统记录账号相关敏感操作。

### API 数据流

#### 智能问答数据流

1. 前端调用 `POST /api/v1/portal/chat-sessions`，提交问题、用户标识和知识库编码。
2. 后端写入问答会话记录。
3. 后端调用 AnythingLLM 适配器，提交问题、知识库标识和上下文参数。
4. AnythingLLM 完成知识检索增强和 RAG 编排，并通过 vLLM OpenAI-compatible 适配器生成答案。
5. vLLM 同步返回完整答案，AnythingLLM 返回答案、来源片段和置信度分数。
6. 后端保存答案、来源文档名称、置信度和耗时。
7. 前端展示答案、来源文档名称和反馈入口。
8. 若置信度低于系统配置的阈值，前端额外展示"仍需人工处理"入口。

#### 申告处理数据流

1. 前端调用 `POST /api/v1/portal/tickets` 创建申告。
2. 后端校验字段并生成申告编号（TK-YYYYMMDD-XXXX）。
3. 后端保存申告、问答上下文和初始状态"待处理"。
4. 后台调用 `GET /api/v1/admin/tickets` 获取待处理列表。
5. 运维人员调用 `PATCH /api/v1/admin/tickets/{id}/status` 更新状态并写入处理记录。
6. 处理完成后，后端可调用 `POST /api/v1/admin/tickets/{id}/knowledge-candidate` 生成知识候选。

#### 知识库发布数据流

1. 管理员调用 `POST /api/v1/admin/knowledge-articles` 创建知识草稿。
2. 管理员调用 `POST /api/v1/admin/knowledge-articles/{id}/submit-review` 提交审核。
3. 审核人（非创建人）调用 `POST /api/v1/admin/knowledge-articles/{id}/review` 审核知识（通过/驳回并填写原因）。
4. 发布人员调用 `POST /api/v1/admin/knowledge-articles/{id}/publish` 将审核通过的知识发布。
5. 后端同步调用 AnythingLLM 适配器同步知识内容和元数据。
6. 后端按知识库配置的 embedding 模型和维度生成或接收切片向量，并写入 pgvector。
7. 后端记录同步状态（成功/失败）和审计日志。
8. 同步失败时，知识状态标记为"同步失败"，管理员可在后台查看错误详情并手动重试。

**注意：** MVP 阶段知识同步采用同步处理。NATS JetStream 异步同步作为后续增强，不在 MVP 范围内。

#### 账号权限数据流

1. 用户调用 `POST /api/v1/auth/login` 登录。
2. 后端校验账号、密码和账号状态。
3. 后端返回令牌（JWT）、用户信息、角色和权限。
4. 前端按权限加载菜单。
5. 后续请求携带令牌访问业务接口。
6. 后端中间件完成认证和授权校验。

## 7. 技术考虑

### 运行环境

- **客户端：** 主流桌面浏览器，建议 Chrome、Edge 最新稳定版。
- **服务端：** PC 或服务器，最低参考配置 4 核 CPU、8GB 内存、100GB 硬盘。
- **网络：** 测试环境可连接互联网；企业环境应支持内网访问。
- **后端：** Go 服务，提供 REST API。
- **前端：** Vue 管理端与门户端。
- **数据库：** MVP 固定采用 PostgreSQL 18，并启用 pgvector 承载系统侧知识向量存储能力。
- **RAG 服务：** AnythingLLM 作为 Docker 内部组件集成，由 `docker-compose.yml` 统一拉起。负责完整 RAG 流程，包括知识导入、切分、向量检索（内部向量库 LanceDB）、RAG 编排和问答。后端通过 `RagClient` 适配层接入，内部地址 `http://anythingllm:3001/api`。
- **AI 推理：** vLLM 作为 Docker 内部组件，通过 AnythingLLM 的 `generic-openai` 提供商接入，OpsMind 后端不直接调用 vLLM。
- **向量配置：** 系统管理员在系统配置中预配置可用的 embedding 模型列表（支持 API 接入和本地部署），每条配置包含模型名称、类型、API 地址或本地路径、向量维度。知识库创建时从该列表中选择模型和维度，同一知识库内所有知识切片必须一致。embedding 模型配置用于 OpsMind 系统侧 pgvector 向量生成，AnythingLLM 内部的 embedding 配置独立管理。
- **向量存储边界：** AnythingLLM LanceDB 负责 RAG 检索，PostgreSQL pgvector 用于 OpsMind 系统侧向量追溯和后续原生检索扩展。两者独立，互不依赖。
- **对象存储：** MVP 采用 MinIO，并通过 S3-compatible 适配层管理申告附件和知识文档原件。

### 设计和实现约束

- 后端优先使用 Go，前端优先使用 Vue。
- MVP AnythingLLM 作为 Docker 内部组件集成，OpsMind 后端通过 `RagClient` 适配层接入，内部地址 `http://anythingllm:3001/api`。
- MVP vLLM 通过 AnythingLLM 的 `generic-openai` 提供商接入，OpsMind 后端不直接调用 vLLM。
- MVP RAG 能力由 AnythingLLM 承担（知识导入、切分、向量检索、RAG 编排），内部向量库使用 LanceDB。PostgreSQL pgvector 仅用于 OpsMind 系统侧向量存储和追溯。
- MVP 每个知识库必须绑定一个 embedding 模型和一个向量维度（用于 pgvector 侧）；发布、切片、同步时禁止写入与知识库配置不一致的向量。
- MVP 对象存储固定通过 S3-compatible 适配层接入 MinIO。
- MVP 运维账号管理只做系统内本地模拟，不对接真实企业账号中心。
- MVP 知识同步采用同步处理，NATS JetStream 异步同步作为后续增强。
- 所有敏感操作需要记录审计日志。
- 模型回答需要保留来源引用和知识命中信息（文档名称），避免无法追溯。
- 文档需覆盖业务流程和 API 数据流，README 需说明项目结构与启动方式。

### 假设和依赖

- 企业可提供基础运维 FAQ、处理方案和账号管理规则。
- vLLM OpenAI-compatible 服务可用，或可部署在独立模型节点上。
- AnythingLLM 作为 Docker 内部组件部署，通过 `docker-compose.yml` 统一拉起，后端通过内部地址 `http://anythingllm:3001/api` 访问。
- PostgreSQL 已启用 pgvector 扩展，用于 MVP 系统侧知识向量存储和后续原生检索扩展。
- embedding 模型和维度由系统管理员在系统配置中预配置；知识库创建时从预配置列表中选择。
- MinIO 服务可用，并提供 S3-compatible API。
- 系统初期以单企业内部使用为主，不考虑复杂多租户隔离。
- 账号创建、冻结、修改等操作在 MVP 阶段只实现系统内本地模拟，不直接对接真实企业账号中心。

### 性能需求

- 常规管理接口响应时间应小于 1 秒。
- MVP 智能问答同步返回完整答案，测试环境中完整回答建议小于 20 秒；首字响应和 SSE 流式输出作为后续增强指标。
- 申告提交、状态更新等核心操作成功率应不低于 99%。
- 列表页面应支持分页查询，默认每页 10 至 20 条。
- 知识库检索应返回 Top K 相关片段，K 值可配置（1-20，默认 5）。

### 可靠性需求

- 模型服务或 RAG 服务不可用时，系统应返回明确提示"当前 AI 服务暂不可用，请提交申告由人工处理"，并引导用户进入申告流程。
- 核心业务数据写入失败时，应返回明确错误并避免重复提交。
- 申告编号生成应唯一（TK-YYYYMMDD-XXXX 格式，XXXX 为当日自增序号）。
- 后台处理状态流转应可追踪、可恢复。
- 系统应记录关键异常日志，便于定位问题。

### 安全性需求

- 后台接口必须登录后访问。
- 权限控制应覆盖菜单和接口。
- 密码应使用 bcrypt 哈希存储，不允许明文保存。
- 敏感操作如冻结账号、发布知识、处理申告应记录审计日志。
- 问答内容、申告内容和知识库内容应按企业内部数据处理。
- 模型提示词和知识检索结果不得泄露系统密钥或内部配置。

### 可维护性需求

- 后端按模块拆分，建议包含认证、用户权限、知识库、问答、申告、日志等模块。
- 前端按门户端和后台管理端组织页面。
- 接口返回结构统一，错误码和错误信息可维护。
- 关键业务逻辑需要单元测试或集成测试覆盖。
- README 需说明项目结构、依赖安装、配置方式和启动步骤。

### 可扩展性需求

- RAG 服务通过 AnythingLLM 适配层接入（Docker 内部组件），后续可在不改业务接口的前提下替换为自建 RAG 方案。
- vLLM 通过 AnythingLLM `generic-openai` 提供商接入，后续替换模型只需修改 AnythingLLM 配置，不改 OpsMind 代码。
- pgvector 只承担系统侧向量存储和扩展检索基础，RAG 检索走 AnythingLLM LanceDB。
- 知识库配置应支持选择 embedding 模型和向量维度，并保证同一知识库内向量模型和维度一致。
- 工单状态机应保留扩展空间，支持后续分派、升级、SLA 等能力。
- 菜单和权限应支持新增模块。
- 系统应预留智能巡检、故障自愈、任务编排、日志分析、告警中枢等扩展模块入口。

### 数据一致性

- 申告状态变更和处理记录写入应保持一致。
- 知识发布状态和 RAG 同步状态应分别记录，避免误认为已生效。
- 知识切片向量写入时应校验知识库 embedding 模型和维度一致性。
- 账号冻结状态应与登录校验实时一致。
- 问答反馈应关联到原始会话和答案。
- 关键写操作建议使用事务保证主表和日志表一致。

### 兼容性要求

- 前端适配主流桌面浏览器。
- API 使用 JSON 数据格式。
- 系统配置应支持开发、测试和生产环境区分。
- 测试环境允许互联网访问，企业部署环境应支持内网运行。

## 9. 风险与待澄清项

### 风险评估

| 风险 | 影响 | 概率 | 应对措施 |
| --- | --- | --- | --- |
| 知识库内容不足 | 问答命中率低 | 高 | 初期聚焦高频 FAQ，建立人工回流机制 |
| 模型回答不准确 | 用户误操作或信任下降 | 中 | 展示来源、提供转人工入口、记录反馈 |
| 本地硬件资源有限 | 推理速度慢或模型无法运行 | 中 | vLLM 可部署到独立模型节点，业务服务通过 OpenAI-compatible 接口调用 |
| 真实账号系统难以对接 | 账号管理范围受限 | 中 | MVP 阶段实现系统内账号模拟 |
| 权限设计不足 | 数据泄露或误操作 | 中 | 采用 RBAC，敏感操作审计 |
| 范围过大 | 课程设计无法按期完成 | 高 | P0 聚焦问答、申告、人工处理、账号和知识库 |

### 待澄清项

- 本地模型实际推理性能需要在目标硬件（4C8GB）上验证，可能影响模型选型。
- AnythingLLM 与 pgvector 的协同边界需要明确：哪些检索由 AnythingLLM 完成，哪些由 pgvector 原生检索承担。
- embedding 模型的可选范围（API 接入的模型提供商、本地可部署的模型）需要在部署前确定。
- 申告附件上传的文件类型和大小限制需要明确。
- 站内消息的具体实现方式（数据库轮询/WebSocket/SSE）需要在 M1 设计阶段确认。
