## 1. 引言

### 1.1 项目背景

企业运维工作存在重复性高、流程性强、知识依赖重、人工响应压力大的特点。一线员工在账号管理、常见问题咨询、故障申告、报障跟进等场景中，需要反复查询制度、知识库和处理流程，导致响应效率低、服务体验不稳定、知识沉淀不足。

本项目基于 AI、RAG 私有知识库和自动化运维能力，建设面向企业运维场景的数字员工系统。系统通过本地化大模型、私有知识库、运维申告门户和后台运营管理能力，辅助或替代人工完成常见咨询、自助处理、申告记录、人工流转和知识库更新等工作。

### 1.2 项目目的

系统目标是为企业提供智能化运维服务入口，降低一线运维人员重复劳动，提高常见问题自助解决率，并形成可持续迭代的运维知识资产。

核心目的包括：

- 提供常见运维问题的自助问答和处理指引。
- 支持无法自动解决的问题进入在线记录或人工处理流程。
- 支持运维账号增删改查等后台维护能力。
- 支持人工处理结果回流知识库，提升后续自动应答质量。
- 为后续扩展智能巡检、故障自愈、任务编排、日志分析、告警中枢等企业级运维能力预留架构空间。

### 1.3 项目特色

- 私有化部署：大模型与知识库优先支持本地部署，适配企业数据安全要求。
- RAG 知识增强：基于企业运维知识库提供可追溯、可维护的问答能力。
- 人机协同闭环：机器人无法处理时流转人工，人工处理结果可沉淀为知识。
- 运维门户一体化：前台服务入口与后台维护能力统一建设。
- 企业级架构取向：后端优先采用 Go，前端优先采用 Vue，架构参考 GVA 项目的前后端分离设计思路。

### 1.4 项目风险

- 知识库质量风险：知识不完整或过期会直接影响问答准确率。
- 大模型幻觉风险：模型可能生成不准确答案，需要答案来源、兜底流转和审核机制。
- 数据安全风险：运维账号、工单记录和企业知识库涉及敏感信息，需要权限与审计。
- 集成复杂度风险：RAG、模型服务、门户系统、账号管理和人工处理流程存在接口协同成本。
- 硬件资源风险：4C8GB 内存环境对本地大模型能力有限，需要合理选择模型规模和推理方案。

### 1.5 产品范围

本期产品范围聚焦运维数字员工 MVP：

- 运维知识库管理。
- 运维智能问答。
- 无法回答时的在线申告记录。
- 人工处理与回访记录。
- 运维账号管理。
- 基础权限、日志和系统配置。

不在本期范围：

- 完整 RPA 桌面自动化编排。
- 生产级故障自愈闭环。
- 大规模多租户商业化计费。
- 复杂告警中枢和日志智能分析平台。

## 2. 综合描述

### 2.1 用户类和特性

| 用户类型 | 用户特性 | 核心诉求 |
| --- | --- | --- |
| 报障人/一线员工 | 对运维流程不熟悉，期望快速获得答案 | 自助查询、提交问题、查看处理结果 |
| 运维人员 | 负责处理账号、故障和申告事项 | 快速定位问题、记录处理过程、回访用户 |
| 知识库管理员 | 负责维护 FAQ、处理方案和知识文档 | 审核知识、更新内容、提升问答质量 |
| 系统管理员 | 负责用户、角色、权限和系统配置 | 账号管理、权限控制、运行监控 |
| 项目评审人员 | 关注课程设计完整性和系统可演示性 | 清晰业务流程、可运行原型、完整文档 |

### 2.2 运行环境

- 客户端：主流桌面浏览器，建议 Chrome、Edge 最新稳定版。
- 服务端：PC 或服务器，最低参考配置 4 核 CPU、8GB 内存、100GB 硬盘。
- 网络：测试环境可连接互联网；企业环境应支持内网访问。
- 后端：Go 服务，提供 REST API。
- 前端：Vue 管理端与门户端。
- 数据库：MVP 固定采用 PostgreSQL 18，并启用 pgvector 承载系统侧知识向量存储能力。
- AI 服务：MVP 采用 vLLM，并通过 OpenAI-compatible 适配层统一模型调用协议。
- RAG 服务：MVP 由 AnythingLLM 负责完整 RAG 流程，包括知识导入、切分、检索增强和问答编排；后端通过 `RagClient` 适配层接入。
- 向量配置：同一个知识库必须使用同一个 embedding 模型和同一个向量维度；创建知识库时必须从系统配置的模型和维度选项中选择，保存后该知识库内知识切片必须一致。
- 对象存储：MVP 采用 MinIO，并通过 S3-compatible 适配层管理申告附件和知识文档原件。

### 2.3 设计和实现上的限制

- 前后端架构参考 Gin-Vue-Admin 的分层设计和权限模型。
- 前端 UI 设计遵循 `docs/prompts/DESIGN-linear.app.md` 中的 Linear Design 约束。
- 后端优先使用 Go，前端优先使用 Vue。
- MVP 问答接口先采用同步返回完整答案，SSE 流式输出作为后续增强能力预留。
- MVP 模型调用固定走 vLLM OpenAI-compatible 适配层，不在业务代码中直接绑定模型私有 API。
- MVP RAG 能力固定通过 AnythingLLM 适配层接入，由 AnythingLLM 承担完整 RAG 流程；PostgreSQL pgvector 用于系统侧向量存储、追溯和后续原生检索扩展。
- MVP 每个知识库必须绑定一个 embedding 模型和一个向量维度；发布、切片、同步时禁止写入与知识库配置不一致的向量。
- MVP 对象存储固定通过 S3-compatible 适配层接入 MinIO。
- MVP 运维账号管理只做系统内本地模拟，不对接真实企业账号中心。
- MVP 异步能力优先满足知识同步和审计留痕，不提前建设完整任务编排平台。
- 所有敏感操作需要记录审计日志。
- 模型回答需要保留来源引用或知识命中信息，避免无法追溯。
- 文档需覆盖业务流程和 API 数据流，README 需说明项目结构与启动方式。

### 2.4 假设和约束（依赖）

- 企业可提供基础运维 FAQ、处理方案和账号管理规则。
- vLLM OpenAI-compatible 服务可用，或可部署在独立模型节点上。
- AnythingLLM 服务可用，并可通过后端 RAG 适配层完成完整 RAG 流程。
- PostgreSQL 已启用 pgvector 扩展，用于 MVP 系统侧知识向量存储和后续原生检索扩展。
- embedding 模型和维度由知识库配置决定；同一知识库下所有知识切片必须使用一致的模型和维度。
- MinIO 服务可用，并提供 S3-compatible API。
- 系统初期以单企业内部使用为主，不考虑复杂多租户隔离。
- 账号创建、冻结、修改等操作在 MVP 阶段只实现系统内本地模拟，不直接对接真实企业账号中心。

## 3. 功能性需求分析

### 3.1 产品概述

运维数字员工系统由前台门户、后台管理、AI 问答服务、知识库服务和基础系统能力组成。前台面向报障人与一线员工，提供自助问答、申告提交和处理结果查看。后台面向运维人员和管理员，提供申告处理、账号管理、知识库维护、用户权限和日志审计。

### 3.2 用户故事与功能清单

#### 3.2.1 用户故事

- 作为报障人，我希望输入运维问题后立即获得处理方案，以便不用等待人工响应。
- 作为报障人，我希望机器人无法回答时可以提交申告记录，以便问题不会中断。
- 作为运维人员，我希望查看待处理申告并记录处理结果，以便统一跟踪问题闭环。
- 作为知识库管理员，我希望将人工处理后的有效方案沉淀到知识库，以便后续自动回答相同问题。
- 作为系统管理员，我希望管理账号、角色和权限，以便保障系统访问安全。

#### 3.2.2 功能清单

| 优先级 | 功能模块 | 功能项 | 说明 |
| --- | --- | --- | --- |
| P0 | 智能问答 | 问题输入、知识检索、答案生成、来源展示 | 核心自助服务能力 |
| P0 | 申告记录 | 无答案兜底、申告提交、状态查看 | 保证问题可流转 |
| P0 | 人工处理 | 待办列表、处理记录、回访记录、状态流转 | 形成服务闭环 |
| P0 | 账号管理 | 运维账号新增、冻结、修改、查询 | 满足课题基础后台要求 |
| P0 | 知识库管理 | FAQ 新增、编辑、审核、发布 | 支撑 RAG 问答质量 |
| P0 | 用户权限 | 登录、角色、菜单权限、接口权限 | 后台管理基础能力 |
| P1 | 数据看板 | 问答量、解决率、人工转派量、知识命中率 | 支持运营分析 |
| P1 | 操作日志 | 登录日志、敏感操作审计、处理日志 | 支持安全追踪 |
| P1 | 模型配置 | 模型地址、密钥、提示词、召回参数配置 | 便于环境适配 |
| P2 | 智能巡检 | 巡检任务配置、巡检报告生成 | 后续企业级扩展 |
| P2 | 故障自愈 | 规则触发、脚本执行、结果回写 | 后续自动化扩展 |
| P2 | 告警中枢 | 告警接入、聚合、分派、闭环 | 后续平台化扩展 |

### 3.3 功能详细说明

#### 3.3.1 智能问答

功能描述：用户在前台门户输入运维问题，系统通过 AnythingLLM 适配层完成知识检索增强和问答编排，并通过 vLLM OpenAI-compatible 适配层生成回答。MVP 问答接口同步返回完整答案，回答应包含处理步骤、注意事项和知识来源。当置信度不足或无匹配知识时，引导用户提交申告。

用户流程图（文字描述）：

1. 用户进入运维门户。
2. 用户输入问题，例如“账号冻结怎么处理”。
3. 前端提交问题到后端问答接口。
4. 后端通过 AnythingLLM 适配层调用知识检索服务获取相关 FAQ 或文档片段。
5. 后端通过 vLLM OpenAI-compatible 适配层调用模型服务生成答案。
6. 前端同步展示完整答案、来源和“仍需人工处理”入口。
7. 用户确认解决或转入申告流程。

界面交互说明：

- 问答输入框支持多行输入和提交按钮。
- 回答区域展示处理步骤、来源、时间和反馈按钮。
- 答案反馈包含“已解决”“未解决”。
- 未解决时展示申告入口，并自动带入原始问题与问答上下文。

验收标准：

- 用户提交问题后，系统能返回答案或明确的兜底提示。
- 答案展示知识来源或命中文档名称。
- MVP 阶段问答接口同步返回完整答案；流式输出不作为本期验收项。
- 无法回答时，用户可以一键进入申告提交。
- 用户反馈结果可被系统记录。

#### 3.3.2 申告记录

功能描述：当机器人无法解决问题时，用户可提交申告记录，填写问题描述、联系方式、影响范围、紧急程度和附件信息。系统生成申告编号并进入待处理状态。

用户流程图（文字描述）：

1. 用户从问答页或门户入口进入申告表单。
2. 系统自动带入原问题和问答上下文。
3. 用户补充影响范围、联系方式和紧急程度。
4. 用户提交申告。
5. 系统生成申告单并展示当前状态。

界面交互说明：

- 表单必填项包括问题标题、问题描述、联系方式、紧急程度。
- 紧急程度可选择低、中、高。
- 提交成功后展示申告编号和状态。
- 用户可按申告编号或联系方式查询处理进度。

验收标准：

- 必填项缺失时不能提交，并提示具体字段。
- 提交成功后生成唯一申告编号。
- 申告状态初始为“待处理”。
- 申告详情中保留原始问答上下文。

#### 3.3.3 人工处理

功能描述：后台运维人员定期查看机器无法处理的申告记录，进行人工干预、处理记录填写、回访结果登记，并在处理完成后将有效方案提交给知识库管理员审核。

用户流程图（文字描述）：

1. 运维人员登录后台。
2. 运维人员查看待处理申告列表。
3. 运维人员打开申告详情，查看用户问题和机器人问答上下文。
4. 运维人员填写处理过程和处理结果。
5. 运维人员登记回访结果。
6. 系统将申告状态更新为“已完成”或“需补充信息”。
7. 运维人员可选择生成知识库候选条目。

界面交互说明：

- 列表支持按状态、紧急程度、创建时间、关键词筛选。
- 详情页展示申告信息、问答上下文、处理记录和状态流转。
- 状态按钮包括“开始处理”“需补充信息”“处理完成”。
- 处理完成时可勾选“沉淀为知识库候选”。

验收标准：

- 运维人员可以查看、筛选、处理申告。
- 每次状态变化均记录处理人和时间。
- 处理完成必须填写处理结果。
- 被标记为知识库候选的记录可进入知识审核流程。

#### 3.3.4 运维账号管理

功能描述：后台提供运维人员账号的新增、冻结、修改、查询能力。课程设计阶段优先管理系统内运维账号，不直接操作真实企业账号中心。

用户流程图（文字描述）：

1. 管理员进入账号管理页面。
2. 管理员新增账号，填写姓名、账号、角色、联系方式和初始密码。
3. 管理员可查询账号并查看详情。
4. 管理员可修改账号基础信息和角色。
5. 管理员可冻结或恢复账号。

界面交互说明：

- 账号列表支持关键词、角色、状态筛选。
- 新增和编辑使用弹窗或独立表单。
- 冻结账号需要二次确认并填写原因。
- 账号状态包括正常、冻结。

验收标准：

- 管理员可以完成账号新增、查询、修改、冻结。
- 账号名不可重复。
- 冻结账号无法登录后台。
- 敏感操作写入审计日志。

#### 3.3.5 知识库管理

功能描述：知识库管理员维护知识库、FAQ 问答对、处理方案和文档资料。每个知识库需要配置 embedding 模型和向量维度，同一知识库内所有知识切片必须保持一致。知识条目支持草稿、待审核、已发布、已停用等状态。已发布知识由 AnythingLLM 完整 RAG 流程检索和编排回答，系统侧使用 pgvector 保存切片向量用于追溯和扩展。

用户流程图（文字描述）：

1. 管理员创建或选择知识库，并确认该知识库的 embedding 模型和向量维度。
2. 管理员新增知识条目。
3. 管理员填写问题、答案、分类、标签和适用范围。
4. 管理员提交审核。
5. 审核通过后发布知识。
6. 系统同步知识到 AnythingLLM，并将同模型同维度的切片向量写入 pgvector。

界面交互说明：

- 知识列表支持分类、状态、关键词筛选。
- 知识库配置支持选择 embedding 模型和向量维度。
- 编辑页支持 FAQ 结构化字段。
- 发布前需要审核。
- 已发布知识修改后需要重新审核。

验收标准：

- 知识条目可新增、编辑、审核、发布、停用。
- 同一知识库下所有向量必须使用相同 embedding 模型和维度。
- 已发布知识可被 AnythingLLM RAG 流程检索。
- 停用知识不再参与问答。
- 每个知识条目保留更新时间和维护人。

#### 3.3.6 用户权限

功能描述：系统提供登录认证、角色管理、菜单权限和接口权限控制。不同角色只能访问授权范围内的功能。

用户流程图（文字描述）：

1. 用户输入账号密码登录系统。
2. 后端校验账号状态和密码。
3. 登录成功后返回访问令牌和用户权限。
4. 前端根据权限展示菜单。
5. 后端对接口请求进行权限校验。

界面交互说明：

- 登录页包含账号、密码和登录按钮。
- 后台菜单按角色动态展示。
- 无权限访问时展示明确提示。
- 管理员可维护角色和权限。

验收标准：

- 正常账号可登录，冻结账号不可登录。
- 不同角色看到不同菜单。
- 未授权接口请求被拒绝。
- 登录和退出记录可追踪。

### 3.4 业务流程

#### 3.4.1 自助问答到人工处理闭环

1. 报障人进入门户并提交问题。
2. 系统检索知识库并生成回答。
3. 报障人反馈“已解决”时，系统记录问答效果。
4. 报障人反馈“未解决”或系统无法回答时，进入申告提交。
5. 运维人员在后台处理申告并回访用户。
6. 处理完成后，运维人员将有效处理方案标记为知识库候选。
7. 知识库管理员审核并发布知识。
8. 新知识参与后续智能问答。

#### 3.4.2 运维账号管理流程

1. 管理员创建运维账号并分配角色。
2. 运维人员使用账号登录后台。
3. 管理员根据人员变化修改或冻结账号。
4. 系统记录账号相关敏感操作。

### 3.5 API 数据流

#### 3.5.1 智能问答数据流

1. 前端调用 `POST /api/v1/portal/chat-sessions`，提交问题、用户标识和知识库编码。
2. 后端写入问答会话记录。
3. 后端调用 AnythingLLM 适配器，提交问题、知识库标识和上下文参数。
4. AnythingLLM 完成知识检索增强和 RAG 编排，并通过 vLLM OpenAI-compatible 适配器生成答案。
5. vLLM 同步返回完整答案，AnythingLLM 返回答案、来源片段和命中信息。
6. 后端保存答案、来源、置信度和耗时。
7. 前端展示答案和反馈入口。

#### 3.5.2 申告处理数据流

1. 前端调用 `POST /api/v1/portal/tickets` 创建申告。
2. 后端校验字段并生成申告编号。
3. 后端保存申告、问答上下文和初始状态。
4. 后台调用 `GET /api/v1/admin/tickets` 获取待处理列表。
5. 运维人员调用 `PATCH /api/v1/admin/tickets/{id}/status` 更新状态并写入处理记录。
6. 处理完成后，后端可调用 `POST /api/v1/admin/tickets/{id}/knowledge-candidate` 生成知识候选。

#### 3.5.3 知识库发布数据流

1. 管理员调用 `POST /api/v1/admin/knowledge-articles` 创建知识草稿。
2. 管理员调用 `POST /api/v1/admin/knowledge-articles/{id}/submit-review` 提交审核。
3. 审核人员调用 `POST /api/v1/admin/knowledge-articles/{id}/review` 审核知识。
4. 发布人员调用 `POST /api/v1/admin/knowledge-articles/{id}/publish` 将审核通过的知识发布。
5. 后端发布知识同步事件，优先通过 NATS JetStream 触发异步同步。
6. Worker 调用 AnythingLLM 适配器同步知识内容和元数据。
7. Worker 按知识库配置的 embedding 模型和维度生成或接收切片向量，并写入 pgvector。
8. 后端记录同步状态、错误信息和审计日志。

#### 3.5.4 账号权限数据流

1. 用户调用 `POST /api/v1/auth/login` 登录。
2. 后端校验账号、密码和账号状态。
3. 后端返回令牌、用户信息、角色和权限。
4. 前端按权限加载菜单。
5. 后续请求携带令牌访问业务接口。
6. 后端中间件完成认证和授权校验。

## 4. 非功能需求分析

### 4.1 性能需求

- 常规管理接口响应时间应小于 1 秒。
- MVP 智能问答同步返回完整答案，测试环境中完整回答建议小于 20 秒；首字响应和 SSE 流式输出作为后续增强指标。
- 申告提交、状态更新等核心操作成功率应不低于 99%。
- 列表页面应支持分页查询，默认每页 10 至 20 条。
- 知识库检索应返回 Top K 相关片段，K 值可配置。

### 4.2 可靠性需求

- 模型服务或 RAG 服务不可用时，系统应提示用户转人工处理。
- 核心业务数据写入失败时，应返回明确错误并避免重复提交。
- 申告编号生成应唯一。
- 后台处理状态流转应可追踪、可恢复。
- 系统应记录关键异常日志，便于定位问题。

### 4.3 安全性需求

- 后台接口必须登录后访问。
- 权限控制应覆盖菜单和接口。
- 密码应使用安全哈希存储，不允许明文保存。
- 敏感操作如冻结账号、发布知识、处理申告应记录审计日志。
- 问答内容、申告内容和知识库内容应按企业内部数据处理。
- 模型提示词和知识检索结果不得泄露系统密钥或内部配置。

### 4.4 可维护性需求

- 后端按模块拆分，建议包含认证、用户权限、知识库、问答、申告、日志等模块。
- 前端按门户端和后台管理端组织页面。
- 接口返回结构统一，错误码和错误信息可维护。
- 关键业务逻辑需要单元测试或集成测试覆盖。
- README 需说明项目结构、依赖安装、配置方式和启动步骤。

### 4.5 可扩展性需求

- 模型服务应通过 vLLM OpenAI-compatible 适配层接入，后续可在不改业务接口的前提下替换模型网关。
- RAG 服务应通过 AnythingLLM 适配层接入，由 AnythingLLM 负责完整 RAG 流程；pgvector 只承担系统侧向量存储和扩展检索基础。
- 知识库配置应支持选择 embedding 模型和向量维度，并保证同一知识库内向量模型和维度一致。
- 工单状态机应保留扩展空间，支持后续分派、升级、SLA 等能力。
- 菜单和权限应支持新增模块。
- 系统应预留智能巡检、故障自愈、任务编排、日志分析、告警中枢等扩展模块入口。

### 4.6 数据一致性

- 申告状态变更和处理记录写入应保持一致。
- 知识发布状态和 RAG 同步状态应分别记录，避免误认为已生效。
- 知识切片向量写入时应校验知识库 embedding 模型和维度一致性。
- 账号冻结状态应与登录校验实时一致。
- 问答反馈应关联到原始会话和答案。
- 关键写操作建议使用事务保证主表和日志表一致。

### 4.7 兼容性要求

- 前端适配主流桌面浏览器。
- API 使用 JSON 数据格式。
- 系统配置应支持开发、测试和生产环境区分。
- 测试环境允许互联网访问，企业部署环境应支持内网运行。

## 5. 风险评估

| 风险 | 影响 | 概率 | 应对措施 |
| --- | --- | --- | --- |
| 知识库内容不足 | 问答命中率低 | 高 | 初期聚焦高频 FAQ，建立人工回流机制 |
| 模型回答不准确 | 用户误操作或信任下降 | 中 | 展示来源、提供转人工入口、记录反馈 |
| 本地硬件资源有限 | 推理速度慢或模型无法运行 | 中 | vLLM 可部署到独立模型节点，业务服务通过 OpenAI-compatible 接口调用 |
| 真实账号系统难以对接 | 账号管理范围受限 | 中 | MVP 阶段实现系统内账号模拟 |
| 权限设计不足 | 数据泄露或误操作 | 中 | 采用 RBAC，敏感操作审计 |
| 范围过大 | 课程设计无法按期完成 | 高 | P0 聚焦问答、申告、人工处理、账号和知识库 |

## 7. MVP 验收清单

- 报障人可以在门户提交问题并获得知识库增强回答。
- 问答链路通过 AnythingLLM 完整 RAG 流程检索和编排知识，并通过 vLLM OpenAI-compatible 适配层同步返回答案。
- 机器人无法解决时，报障人可以提交申告。
- 运维人员可以在后台处理申告并登记回访结果。
- 管理员可以维护本地模拟运维账号并冻结账号。
- 知识库管理员可以维护 FAQ 并发布到问答流程。
- 知识发布后可以完成 AnythingLLM 同步状态记录，系统侧 pgvector 向量满足同知识库同模型同维度约束，失败时可查看错误信息。
- 系统具备登录、角色权限、接口鉴权和基础审计日志。
- 文档包含业务流程、API 数据流、项目结构和启动说明。
